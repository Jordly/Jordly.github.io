## 一、文件安全与备份

### 1.1 数据丢失惨案（2026年6月23-26日）
- **问题**：连续3天的修改只在本地，没有推送到GitHub。后来本地文件被层层覆盖，所有修改全部丢失。
- **根因**：没有养成"改完就推送"的习惯，以为本地保存就安全了。
- **解决方案**：
  - 每次修改完成后，**立即 git commit + git push**
  - 设置工作日 17:55 自动备份（双保险）
  - 绝不依赖"回头再推"，因为可能会忘记
- **教训**：**GitHub云端备份是唯一可靠的安全网，本地随时可能因为各种原因丢失。**

### 1.2 缓存导致"改了但没生效"
- **问题**：修改了文件推送到GitHub后，刷新页面没有任何变化。
- **根因**：index.html 中的版本号 `?v=xxx` 没有更新，Cloudflare CDN 和浏览器缓存都认为文件没变。
- **解决方案**：修改 app.js / style.css 后，**必须同步更新 index.html 中的版本号**。
- **教训**：版本号不是装饰，是防缓存的关键机制。

## 二、代码修改规范

### 2.1 CSS `!important` 优先级陷阱
- **问题**：JS设置 `style.display='none'` 但元素仍然显示。
- **根因**：CSS 中的 `.some-class { display: flex !important }` 优先级高于 JS 内联 style。
- **解决方案**：
  - 尽量不用 `!important`
  - 如果必须用，JS中也要用 `setProperty('display','none','important')`
  - 显隐控制统一走 JS 内联 style，不要在 CSS 里固定 display
- **日期**：2026-07-01（筛选栏修复，反复3轮才找到根因）

### 2.2 JS函数重复定义导致功能异常
- **问题**：修改了一个函数，但功能完全没变化。
- **根因**：app.js 中有**3个同名函数**，JS 中最后一个定义会覆盖前面的，所以旧版函数生效。
- **解决方案**：
  - 添加新函数前先搜索是否有同名函数
  - 有重复就全部统一替换
  - 绝不要在文件不同位置定义同名函数
- **日期**：2026-07-01（`toggleAdvancedFilter()` 有3个重复定义）

### 2.3 遗漏修改导致功能不完整
- **问题**：改了代码但刷新后功能只对了一半。
- **根因**：只改了初始化逻辑的**1处**（如 from=login 路径），遗漏了 `checkLogin()` 后的**第2处**。
- **解决方案**：修改初始化逻辑时，**搜索所有相关位置**，确保全都改了。
- **日期**：2026-07-01（刷新保留当前模块，改了1处漏了1处）

### 2.4 复制文件遗漏
- **问题**：prototype/ 里的文件改了，但线上没变化。
- **根因**：docs/ 目录也需要同步复制一份。Cloudflare Pages 部署的是仓库根目录的 docs/。
- **解决方案**：修改 prototype/ 源文件后，必须复制到 docs/，两个目录都提交推送。
- **注意**：cloudbase-sync.js 只在 docs/ 里需要，prototype/ 不需要。

## 三、安全漏洞

### 3.1 任意账号密码都能登录（致命漏洞）
- **问题**：随便输入账号密码都能登录，还被自动赋予超级管理员权限。
- **根因**：`doLogin()` 不验证密码，找不到用户就自动创建假用户并给最高权限。
- **解决方案**：
  - 必须验证密码
  - 用户不存在直接报错
  - 密码错误直接报错
  - 只有内置默认用户可以兜底
- **日期**：2026-07-01

## 四、排查问题的正确方法

### 4.1 盲目改代码 vs 先分析根因
- **反面案例**（2026-07-03）：弹窗不显示，连续改了3次代码都是错的（加唯一ID→改成querySelector→...），每次都猜错根因。
- **正确做法**：
  1. 先检查浏览器控制台有没有报错
  2. 排查CSS类名是否匹配
  3. 检查DOM元素是否被正确创建（但不可见）
  4. 用户提供的线索非常关键（如"点击后指针变箭头"）
- **核心原则**：**排查超过3轮没进展 → 停下来，告诉用户实际情况，不能无限消耗积分。**

### 4.2 弹窗不显示的常见原因
- CSS类名不匹配（CSS定义 `.sd-confirm-show` 但JS用了 `.sd-prompt-show`）
- `opacity: 0` 且没有触发显示类
- 弹窗被其他元素遮挡（z-index问题）
- **教训**：弹窗创建成功但看不见 → 大概率是CSS类名问题。

## 五、与模型协作的经验

### 5.1 用户反感的行为（红线）
1. 重复犯错（尤其已经明确告知过的信息）
2. 把责任推给用户（"请确认""您再试试"等）
3. 承诺不兑现
4. 让用户做本该模型内部验证的工作（清缓存、开控制台等）
5. 遇到专业术语不先解释就直接执行
6. 不理解需求不先确认，就直接开始执行
7. 功能没实现，却改提示信息/图标来蒙混
8. 改下一个问题时，私自改了原来不需要改的内容
9. 排查问题时陷入无限思考，积分一直消耗但没实质进展
10. 私自改动用户未要求的地方
11. **改某一板块时，绝不能影响其他板块**

### 5.2 正确的配合流程
1. 用户提需求 → **大白话复述一遍**，确认理解对了
2. 给方案 → 告诉用户"我会改X、Y两个文件，不影响其他功能"
3. 用户确认 → 再动手
4. 执行 → 每完成一步，告诉用户结果
5. 完成后 → 自己先验证功能，再告诉用户可以测试
6. 有问题 → 先承认错误，告诉用户"我哪里做错了，我会怎么改"

### 5.3 涉及UI/功能的任何修改，必须先告知
- **任何可能影响系统界面或功能的操作，必须先提醒用户，得到明确确认后才能执行。**
- 不能因为"我觉得这里也可以优化"就私自改动。

## 六、弹窗与UI开发经验

### 6.1 不要用浏览器原生弹窗
- `alert()` / `confirm()` / `prompt()` 在不同设备上样式统一丑陋
- 必须用自定义弹窗（`showConfirmModal` / `showPromptModal` / `showSelectModal`）
- 弹窗要用同一个CSS类名控制显示/隐藏（`sd-confirm-show`）

### 6.2 CSS类名规划
- 统一前缀：`sd-`（CS CloudHub系统弹窗）
- 显示/隐藏统一用 `sd-confirm-show`（不要每个弹窗类型各搞一个类名）
- confirm 弹窗 footer 用 `sd-confirm-footer`
- prompt 弹窗 footer 用 `sd-prompt-footer`
