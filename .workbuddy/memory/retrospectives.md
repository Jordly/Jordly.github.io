## 2026-06-07

### 犯的错误

- **Jordly 原话**：截图中红框内的编辑按钮，不生效
- **我的做法**：追加了 editProject() 函数到 project-import-export.js，但 app.js 中 onclick 生成代码有误
- **根因**：app.js 第2403行，编辑按钮 HTML 被包在普通字符串（单引号）中，${p.id} 未解析。onclick 实际生成 editProject('${p.id}')，p 未定义导致报错
- **改进措施**：在模板字符串嵌套场景下，内层字符串必须使用模板字符串（反引号）才能让 ${...} 正确解析。以后写 HTML 生成代码时，检查变量插值是否在正确的字符串类型中

#### 好的经验
- 将大功能模块拆分为独立 JS 文件（goal-management.js），避免直接修改 app.js 大文件导致的字符串匹配失败问题
- 独立文件在 index.html 中后加载，可覆盖 app.js 中的同名函数，实现无痛升级

#### 犯的错误
- **我的做法**：用 Python 脚本在 app.js 的 showProjectDetail() 模板字符串中插入责任边界调用
- **根因**：搜索模板字符串结束位置（``）时，未验证上下文，导致插入在了错误位置（第3357行，在模板字符串开始之前）

## 2026-06-07 复盘（CloudBase 云端同步修复专题）

### 一、犯的错误

#### 错误1：给了用户错误的域名后缀，浪费大量时间
- **Jordly 原话**："还是显示：无法找到 cscloudhub-...apptcloudbase.com 的 DNS 地址"
- **我的做法**：我之前猜的域名后缀是 `apptcloudbase.com`，实际正确的是 `app.tcloudbase.com`，导致用户反复测试 DNS 找不到
- **根因**：我没有让用户去控制台确认真实域名，而是自己瞎猜了一个格式，猜错了
- **改进措施**：涉及到域名、URL、地址这类信息，**必须先让用户从控制台复制真实内容**，不能靠猜。URL 相关配置，看到真实截图再填，不提前瞎写

#### 错误2：zip 包结构错误，反复重新打包
- **现象**：用户第一次上传 zip 后云函数报错"找不到入口文件"
- **根因**：第一次打包时 zip 内包含了子目录 `cloudbase-api/`，云函数找不到根目录的 `index.js`
- **改进措施**：打包后必须用 `python -c "import zipfile; z=zipfile.ZipFile(...); print(z.namelist())"` 验证文件在根目录，再交给用户

#### 错误3：云函数代码有 bug，用户测试失败
- **现象**：用户新增项目后提示"云端同步失败"
- **根因**：云函数中 `save` 操作直接用 `.doc('singleton').set()`，但文档不存在时会报错；应该先 `update`，失败后再 `add`
- **改进措施**：写云函数代码时，必须考虑"文档不存在"的场景，用 try-catch 处理

#### 错误4：没有在推送前让用户验证云函数是否部署成功
- **现象**：用户多次上传 zip 后，我不知道是否成功，直到用户告诉我"还是失败"
- **根因**：我没有明确要求用户在每次上传后确认"部署成功"提示
- **改进措施**：涉及用户操作云控制台的，每一步都必须明确要求用户**截图或文字确认成功**，再进入下一步
